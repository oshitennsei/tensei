## Context

New greenfield Chrome extension. No existing code. The spec recommends: Manifest V3, TypeScript, React + Tailwind, Vite, side panel API, Dexie.js. The scaffold change establishes project structure and build infrastructure only — no business logic. All subsequent changes (storage layer, LLM client, ingestion, etc.) drop into the `extension/src/lib/` stubs this scaffold creates.

The primary constraint is the **Manifest V3 service worker 30-second idle limit** (spec §21). The design must ensure long-running work (future ingestion, compression) runs in the side panel page context, not the service worker.

## Goals / Non-Goals

**Goals:**
- Loadable, hot-reloadable Chrome extension from day one
- Side panel as main interaction surface, openable from toolbar
- Service worker that stays minimal (lifecycle + message routing only)
- Folder structure that maps exactly to spec §13 layout
- `host_permissions` scoped correctly from the start so future changes don't touch the manifest
- Production build produces a zip-ready dist

**Non-Goals:**
- Any UI beyond the structural shell (no components, no routing beyond a stub)
- IndexedDB schema (separate change)
- LLM client (separate change)
- Content script behavior (stub only — logic comes with paste mode change)
- Popup business logic (opens side panel, nothing more)

## Decisions

### Build tool: `vite-plugin-web-extension` over `crxjs`

`crxjs` has better HMR but its v2 (MV3-compatible) has been in RC for over a year with sporadic maintenance. `vite-plugin-web-extension` is actively maintained, handles manifest generation from `package.json`, and supports multiple entry points (sidebar, popup, background, content scripts) cleanly. The tradeoff is slightly slower HMR (full reload on background changes), acceptable for a side-panel-focused app where most work is in the page context.

### React + Tailwind over vanilla TS + Web Components

The spec offers both options. React wins here because: (a) the UI is complex — multi-character sessions, streaming output, settings panels, BTS/performance modes; (b) future contributors expect React; (c) Tailwind's purge keeps bundle size reasonable. Vanilla Web Components would require significant framework work that distracts from the product.

Bundle size concern: mitigated by Vite's tree-shaking and the fact that the extension only needs to load on user intent (side panel open), not on every page.

### Side panel over popup for main UI

`chrome.sidePanel` (Chrome 114+) gives a persistent page context that survives for the session — critical for hosting long-running sub-agents, compression jobs, and streaming LLM responses. The popup closes on blur and has the same Manifest V3 service worker limitations. The popup entry point is kept but its sole job is to open the side panel.

### Service worker scope: message routing only

Service worker handles: `chrome.runtime.onInstalled`, `chrome.runtime.onMessage` (route to side panel), `chrome.alarms` (future periodic checks). Nothing else. This avoids the 30-second idle kill for any real work. The side panel page context hosts everything else.

### `host_permissions` set at scaffold time

Permissions required by future features must be declared in the manifest. Setting them now avoids manifest churn across changes. Required:
- `https://raw.githubusercontent.com/*` — whitelist fetch
- User LLM endpoints: cannot be hardcoded; use `<all_urls>` scoped with a user-provided allowlist at runtime, OR list known providers (OpenAI, Anthropic, Google). Decision: use `<all_urls>` since Ollama/custom endpoints are user-defined. This is disclosed prominently in first-run UI.

### Folder structure mirrors spec §13 exactly

```
extension/
  manifest.json        (generated by vite-plugin-web-extension from package.json)
  src/
    background/        index.ts — service worker entry
    content/           index.ts — content script stub
    popup/             index.tsx, index.html
    sidebar/           index.tsx, index.html — main surface
    lib/
      storage/         index.ts stub
      llm/             index.ts stub
      agents/          index.ts stub
      memory/          index.ts stub
      retrieval/       index.ts stub
      ingestion/       index.ts stub
      content-safety/  index.ts stub
      persona/         index.ts stub
    prompts/           (empty dir, .gitkeep)
  tests/               (empty dir, .gitkeep)
```

## Risks / Trade-offs

- **`<all_urls>` permission is aggressive** → Mitigated by: (1) Chrome warns users at install; (2) first-run UI explains exactly what it's used for; (3) this is consistent with spec's privacy-first disclosure approach. Narrower alternatives (explicit list of providers) would break Ollama/custom endpoints without workarounds.

- **`vite-plugin-web-extension` generates the manifest** → The `manifest.json` in `src/` becomes the source of truth, not `dist/`. Developers must edit `src/manifest.json`, not touch `dist/`. Documented in README.

- **React bundle in extension** → Side panel initial load ~150-200KB (React + Tailwind) after minification, which is fine for a side panel that the user explicitly opens. Not injected into every page.

## Open Questions

- **Extension ID for development**: Chrome assigns a stable ID for published extensions; dev ID changes each load unless we fix it via a key in manifest. Low priority for Phase 1 (self-use), worth addressing before any alpha distribution.
- **Minimum Chrome version**: `chrome.sidePanel` requires Chrome 114+. Acceptable for now; document in README.
